# 信号页运行历史改进设计方案

## 当前问题

1. **显示"未知时间"** - history 中旧格式数据没有时间戳字段，新格式记录需要等待新运行产生
2. **缺少翻页功能** - 当历史记录较多时，需要分页展示
3. **详情缺少运行结果** - 详情弹窗只显示时间、时长、状态，缺少推荐币种、特征等实际结果
4. **交互不稳定** - 点击记录时弹窗可能意外关闭

## 设计目标

运行历史应该让用户：
- 快速浏览每条运行记录的时间、动作、状态
- 翻页查看更多历史
- 点击单条记录查看完整运行结果（推荐币种、候选排行、训练产物等）
- 稳定地打开和关闭详情弹窗

---

## 数据来源分析

### 当前 history 数据结构（后端）

```python
# 新格式（已修改）
history[action] = [
  {
    "started_at": "2026-04-23T14:30:00Z",
    "finished_at": "2026-04-23T14:30:25Z",
    "duration_seconds": 25,
    "status": "succeeded",
    "message": "研究训练已完成..."
  },
  ...
]
```

### 运行结果数据来源

| 动作类型 | 结果数据来源 | API 端点 |
|---------|-------------|----------|
| training | 最新训练产物、模型版本 | `/api/control/research` → `latest_training` |
| inference | 候选排行、研究报告 | `/api/control/signals/research` → `candidates` |
| pipeline | 完整流水线结果（训练+推理+信号） | 综合以上 |

**问题**：当前 history 只存储了运行元数据，没有关联具体运行结果。运行结果是实时数据，不按历史记录存储。

---

## 解决方案

### 方案一：历史记录关联运行结果快照

**思路**：每次运行完成时，把关键结果（推荐币种、top候选、模型版本）存入 history 记录。

**优点**：
- 详情弹窗可以直接展示历史结果
- 不需要额外 API 调用

**缺点**：
- history 数据体积变大
- 需要修改后端存储逻辑

**数据结构**：
```python
run_record = {
  "started_at": "...",
  "finished_at": "...",
  "duration_seconds": 25,
  "status": "succeeded",
  "message": "...",
  "result_snapshot": {
    "recommended_symbol": "BTCUSDT",
    "recommended_strategy_id": 1,
    "top_candidates": ["BTCUSDT", "ETHUSDT", "SOLUSDT"],
    "model_version": "balanced_v3",
    "signal_count": 10,
  }
}
```

### 方案二：详情弹窗实时查询当前结果

**思路**：详情弹窗不展示历史结果，而是跳转到对应页面查看当前最新结果。

**优点**：
- 简单，不修改后端存储
- 结果始终是最新数据

**缺点**：
- 无法回看历史运行的具体结果
- 用户需要手动去其他页面查找

## 分页设计

### 参数
- 每页显示：5 条记录
- 每个动作类型独立分页
- 使用前端分页（数据量不大）

### 交互
- 点击"上一页/下一页"按钮切换
- 显示当前页码和总页数
- 页码超出范围时隐藏按钮

---

## 弹窗交互稳定性

### 问题分析
当前 FullScreenModal 点击背景遮罩会关闭弹窗，可能导致用户意外点击关闭。

### 解决方案
1. 增加遮罩点击确认：点击遮罩时弹出确认提示"确定要关闭吗？"
2. 或者：移除遮罩点击关闭，只允许点击关闭按钮关闭
3. 详情弹窗使用独立的关闭按钮，避免误操作
