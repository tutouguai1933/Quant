# 前端重构规划与设计

## 1. 当前问题分析

### 1.1 信息密度问题

**首页（page.tsx）：**
- HomeWorkbenchGrid 包含 6 个大卡片，每个卡片都有 drawer 展开
- 大量重复信息：推荐、研究、执行、风险、下一步、回看
- MetricGrid 只显示 4 个简单指标，占用大量空间
- HomePrimaryActionSection 重复展示成功/执行/异常链路

**策略页（strategies/page.tsx）：**
- StrategiesFocusGrid 包含 4 个大卡片（执行器、候选、执行模式、账户）
- 每个卡片都有 4 个 digests + drawer 详情
- 大量仅作提示的文本："先确认..."、"当前还没有..."
- 配置卡片占用大量空间但使用频率低

**任务页（tasks/page.tsx）：**
- 文件过大（27187 tokens），包含大量冗余组件
- 恢复清单、控制矩阵、运行窗口等多个大块重复展示状态
- 自动化状态卡片堆叠过多

### 1.2 用户动线问题

**核心流程不清晰：**
- 研究 → 评估 → 策略 → 执行 → 监控 的主线被大量卡片淹没
- 关键操作入口分散在多个 drawer 和 section 中
- 用户需要点击多次才能找到核心功能

**导航混乱：**
- 页面间跳转逻辑复杂
- 仲裁动作、建议动作、下一步动作等概念重叠
- 缺少明确的"当前在哪一步"的指示

### 1.3 组件复用问题

**重复组件过多：**
- 多个页面都有自己的 PrimaryActionSection
- 状态展示逻辑分散在各个页面
- 缺少统一的状态卡片组件

## 2. 重构目标

### 2.1 信息密度提升

- 移除所有仅作提示的空卡片
- 将 drawer 详情直接内联到主页面（按需折叠）
- 精简重复信息，每条信息只展示一次
- 提升首屏信息量，减少点击次数

### 2.2 用户动线优化

**明确核心流程：**
```
研究工作台 → 评估中心 → 策略判断 → 执行派发 → 任务监控
```

**每个页面的核心职责：**
- 首页：快速决策入口，显示当前状态和下一步动作
- 研究页：因子分析、模型训练、候选生成
- 评估页：候选评估、门控判断、优先队列
- 策略页：策略判断、信号生成、执行派发
- 任务页：自动化监控、异常处理、恢复操作

### 2.3 组件复用策略

**统一状态卡片：**
- StatusCard：显示单个状态（执行器、账户、自动化等）
- ActionCard：显示可执行动作
- MetricCard：显示指标数据

**统一布局组件：**
- PageLayout：统一页面布局
- SectionHeader：统一区块标题
- InfoGrid：统一信息网格

## 3. 设计方案

### 3.1 首页重构

**目标：**从 6 个大卡片 + 4 个指标 → 精简为 3 个核心区块

**新布局：**
```
┌─────────────────────────────────────────┐
│ PageHero: 当前推荐 + 下一步动作          │
├─────────────────────────────────────────┤
│ StatusBar: 研究 | 执行 | 风险 | 自动化   │
├─────────────────────────────────────────┤
│ QuickActions: 核心操作按钮              │
└─────────────────────────────────────────┘
```

**移除：**
- HomeWorkbenchGrid（6 个大卡片）
- MetricGrid（4 个简单指标）
- HomePrimaryActionSection（重复信息）

**保留并优化：**
- PageHero：显示当前推荐标的和下一步动作
- StatusBar：一行显示 4 个核心状态
- QuickActions：3-5 个核心操作按钮

### 3.2 策略页重构

**目标：**从 4 个大卡片 + drawer → 精简为 2 列布局

**新布局：**
```
┌──────────────────────┬──────────────────────┐
│ 左列：策略判断        │ 右列：执行状态        │
├──────────────────────┼──────────────────────┤
│ - 当前推荐候选        │ - 执行器状态          │
│ - 策略评分            │ - 最近信号            │
│ - 推荐动作            │ - 最近订单            │
│ - 候选队列（前3个）   │ - 账户摘要            │
├──────────────────────┴──────────────────────┤
│ 执行动作区：启动/暂停/派发                   │
└─────────────────────────────────────────────┘
```

**移除：**
- StrategiesFocusGrid（4 个大卡片）
- 所有 drawer 详情
- 配置卡片（移到设置页）

**保留并优化：**
- 策略判断直接展示
- 执行状态内联显示
- 控制动作统一到底部

### 3.3 任务页重构

**目标：**从 27187 tokens → 精简到 10000 tokens 以内

**新布局：**
```
┌─────────────────────────────────────────┐
│ 自动化状态：模式 | 状态 | 最近一轮      │
├─────────────────────────────────────────┤
│ 当前阻塞：显示最重要的 1-2 个阻塞       │
├─────────────────────────────────────────┤
│ 控制动作：恢复/暂停/切换模式            │
├─────────────────────────────────────────┤
│ 最近任务：表格展示最近 10 条            │
└─────────────────────────────────────────┘
```

**移除：**
- 恢复清单详情（只显示阻塞项）
- 运行窗口详情（只显示关键信息）
- 大量说明文本

**保留并优化：**
- 自动化状态一行显示
- 阻塞项突出显示
- 控制动作简化为 3-5 个按钮

### 3.4 评估页重构

**目标：**突出候选队列和评估决策

**新布局：**
```
┌─────────────────────────────────────────┐
│ 当前推荐：标的 | 评分 | 推荐动作         │
├─────────────────────────────────────────┤
│ 候选队列：表格展示所有候选（可排序）     │
├─────────────────────────────────────────┤
│ 评估决策：门控状态 | 阻塞原因            │
└─────────────────────────────────────────┘
```

## 4. 组件设计

```tsx
<StatusBar
  items={[
    { label: "研究", value: "完成", status: "success" },
    { label: "执行", value: "运行中", status: "running" },
    { label: "风险", value: "0", status: "safe" },
    { label: "自动化", value: "dry-run", status: "active" },
  ]}
/>
```

```tsx
<StatusCard
  title="执行器状态"

/>
```

```tsx
<ActionCard
  title="运行自动化周期"
  description="执行一轮训练、推理、执行和复盘"

  variant="primary"
/>
```

## 6. 验收标准

### 6.1 信息密度
- 首页首屏信息量提升 50%
- 点击次数减少 30%
- 页面代码量减少 40%

### 6.2 用户动线
- 核心流程清晰可见
- 关键操作 2 次点击内可达
- 当前位置明确标识

### 6.3 性能
- 首页加载时间 < 2s
- 页面切换流畅
- 无明显卡顿

## 7. 风险与注意事项

### 7.1 兼容性
- 保持 API 接口不变
- 保持路由结构不变
- 保持核心功能不变

### 7.2 渐进式重构
- 先重构首页，验证效果
- 再逐步重构其他页面
- 每个阶段都要测试

### 7.3 回滚方案
- 保留旧组件代码
- 使用 feature flag 控制
- 出问题可快速回滚
