# Qlib 单币择时研究工厂 Design

## 1. Summary

围绕 `BTCUSDT / ETHUSDT / SOLUSDT / DOGEUSDT` 建一套可复现的单币择时研究工厂，用 `Qlib` 负责数据、训练、回测、评估和排行榜，用规则层 + ML 层双层门控筛出值得进入 `dry-run` 的候选策略。

这样做的意义是把“研究是否有效”和“能不能进真实验证”分开，先让研究流程稳定，再逐步把稳定结果送到执行层。

## 2. Goals / Non-goals

### Goals

- 做一条完整的 `数据 -> 特征 -> 标签 -> 训练 -> 回测 -> 评估 -> 候选入池` 研究闭环。
- 研究范围只覆盖 `BTCUSDT / ETHUSDT / SOLUSDT / DOGEUSDT` 四个币。
- 持有周期聚焦 `1-3 天`，不做高频。
- 第一层规则门负责过滤明显不适合买卖的时段。
- 第二层 `Qlib` 模型负责给出 `buy / sell / watch` 倾向。
- 输出统一排行榜，明确哪些因子组合和策略模板值得进入 `dry-run`。

### Non-goals

- 不做多币轮动排名与组合优化。
- 不做高频、做市、套利类策略。
- 不在这个阶段追求复杂深度学习模型。
- 不把 `Qlib` 直接变成实盘执行器。
- 不扩展到 `Lean / vn.py / OpenClaw`。

## 3. Current State & Constraints

### Current state

- 控制平面、市场页、图表页、策略页、任务页已经可用。
- `Freqtrade` 的 `dry-run` 与最小 `live` 验证已经跑过，执行层链路可用。
- `Qlib` 最小训练与推理闭环已经存在，页面也已经能展示研究状态、研究分数和研究解释。
- 当前研究输出更偏“最小可运行”，还不是完整的择时研究工厂。

### Constraints

- 开发环境以 `WSL + quant conda 环境` 为主，部署环境以阿里云为主。
- 当前项目强调低风险、可验证和可复现。
- 依赖安装、lockfile 变更仍然需要单独批准。
- 研究结果必须能说明为什么允许或不允许进入 `dry-run`。

## 4. Requirements

### P0

- 支持四个固定币种的单币择时研究。
- 支持双层门控：
  - 规则层先判断当前市场是否允许进入交易候选区。
  - ML 层再给出 `buy / sell / watch` 倾向。
- 支持时间序列切分，不允许随机切分训练集与验证集。
- 支持统一回测评估，至少输出：
  - 收益
  - 最大回撤
  - Sharpe
  - 胜率
  - 换手
- 支持研究结果排行榜，并明确“是否允许进入 dry-run”。

### P1

- 支持类别不平衡处理。
- 支持特征筛选。
- 支持时间序列交叉验证。
- 支持超参数搜索。

### P2

- 支持多个策略模板并行比较。
- 支持自动生成可读研究报告摘要。

## 5. Design

### Overall approach

整体采用“双层门控 + 统一评估”的研究工厂：

- 数据层：准备 `1h` 与 `4h` 两个周期的市场样本。
- 因子层：先做少量稳定因子，不追求大而全。
- 规则层：判断趋势、波动和量能是否允许进入候选区。
- ML 层：用 `Qlib` 模型输出 `buy / sell / watch` 分数。
- 回测层：按 `1-3 天` 持有周期评估策略。
- 筛选层：只有通过统一门槛的结果，才允许进入 `dry-run` 候选池。

### Key decisions

- 先做单币择时，不做币种轮动。这样更容易判断信号是否稳定。
- 先做中低频，不做高频。个人开发者更容易拿到稳定反馈。
- 先用少量因子，不堆特征数量。先要稳定，再要复杂。
- 先用规则层 + `LightGBM` 一类稳定模型，不直接上复杂深度学习。
- 回测与评估必须成为固定出口，而不是临时分析。

### Candidate factor groups

- 趋势因子：
  - `EMA20 / EMA55`
  - 价格相对均线位置
- 动量因子：
  - 最近 `N` 根收益
  - 突破强度
- 波动因子：
  - ATR
  - 最近波动率
- 量能因子：
  - 成交量相对均值倍数
- 震荡辅助：
  - RSI 或同类区间指标

### Candidate strategy templates

- `trend_breakout_timing`
  - 趋势成立后，等待突破确认再考虑买入。
- `trend_pullback_timing`
  - 趋势成立后，等待回踩完成再考虑买入。

两套模板都用同一批研究指标和统一回测口径，便于横向比较。

### Label design

第一版标签只围绕 `1-3 天` 持有周期定义：

- `buy`
  - 在未来窗口内达到目标收益，且未先触发止损。
- `sell`
  - 当前持有逻辑失效，或未来窗口先触发止损。
- `watch`
  - 没有足够优势，不进入交易。

标签先追求清晰和可解释，不追求复杂。

### Entry gate to dry-run

候选结果必须通过四道门：

- 样本内不过关：淘汰。
- 样本外不过关：淘汰。
- 回撤或换手超出上限：淘汰。
- `dry-run` 验证不稳定：淘汰。

只有都通过，才允许进入真实执行前的 `dry-run` 阶段。

### Alternatives

#### A. 纯规则优先

优点：快、容易解释。  
缺点：上限较低，后续容易进入人工调参。

#### B. 纯 ML 优先

优点：更接近最终目标。  
缺点：更容易混淆数据问题、模型问题和执行问题。

#### C. 规则 + ML 双层门控

优点：更稳，也更适合个人开发者。  
缺点：比单一路线多一层设计工作。

推荐使用 C。

## 6. Acceptance Criteria

- 系统能针对四个固定币种输出统一的单币择时研究结果。
- 研究结果至少包含：
  - 因子摘要
  - 模型分数
  - `buy / sell / watch`
  - 回测指标
  - 是否允许进入 `dry-run`
- 训练、验证、测试切分明确，且是时间序列切分。
- 页面或接口能看到统一的候选排行榜。
- 至少两套策略模板能在相同口径下比较。
- 研究结果能说明为什么通过或没通过 `dry-run` 门槛。

## 7. Validation Strategy

- Worker 测试：
  - 训练、推理、特征、标签测试通过。
- API 测试：
  - 研究结果读取、候选列表、排行榜接口测试通过。
- 页面验证：
  - 能看到候选结果、研究解释、门槛状态。
- 回测验证：
  - 至少能稳定输出统一指标，不报错、不缺字段。

## 8. Risks & Rollback

### Risks

- 因子过多导致过拟合。
- 标签设计不合理，导致回测看起来好但实盘没意义。
- 时间切分不严格会带来数据泄漏。
- 研究结果和执行门槛分离不清，会让“看起来能做”和“实际能做”混淆。

### Rollback

- 如果双层门控过重，可先退回“规则层 + 最小模型”。
- 如果模型层效果不稳定，可暂时只保留规则择时，继续做研究工厂骨架。
- 如果回测链不稳定，先固定单一模板，再恢复多模板比较。
