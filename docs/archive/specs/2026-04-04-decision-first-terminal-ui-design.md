# Quant 决策优先交易终端设计

## 1. Summary

把当前偏后台列表式的前端，重构成“专业交易终端外观 + 决策优先动线”的统一界面，让用户先看到研究结论和下一步动作，再进入图表、执行和结果页。

## 2. Goals / Non-goals

### Goals

- 首页、信号页、单币页、策略页统一成双栏终端结构。
- 页面优先回答“现在该看什么、该做什么”，减少竖向堆叠和重复说明。
- 研究结果、候选筛选、回测摘要和执行动作使用同一套视觉层级。
- 保留现有功能和接口，不改变研究、执行和鉴权主链路。
- 让市场页、单币页、策略页之间的跳转更顺，减少用户迷路。

### Non-goals

- 本轮不重做后端接口和研究逻辑。
- 本轮不引入新的前端依赖和新的设计系统框架。
- 本轮不先改风险、任务、余额、订单、持仓的深层业务逻辑。
- 本轮不处理 OpenClaw、Lean、vn.py 扩展位。

## 3. Current State & Constraints

### Current State

- 当前前端已有研究结果、候选、图表、策略控制和账户结果页面。
- 当前问题不是功能缺失，而是布局偏纵向堆叠，主次不够清晰。
- 决策信息、图表信息、动作按钮和解释文案分散在多个面板里，容易拥挤。

### Constraints

- 本地端口继续使用 `9011/9012/9013`。
- 不自动安装新依赖，不修改依赖声明。
- 修改页面后必须做真实页面验证，不只跑测试。
- 保持现有 UTF-8 文件编码，不改坏中文。

## 4. Requirements

### P0

- 首页改成“决策入口页”，先展示推荐候选、执行状态、下一步动作。
- 信号页改成“左侧研究候选，右侧统一研究报告”的双栏结构。
- 单币页改成“左侧判断与动作，右侧主图”的终端布局。
- 策略页改成“左侧推荐执行，右侧执行器状态和操作”的布局。
- 全站主页面统一使用同一套摘要条、状态条和双栏主区。

### P1

- 统一卡片层级，减少卡片套卡片。
- 统一空状态和失败状态文案，直接告诉用户下一步做什么。
- 统一主要按钮和次要按钮的层级，不让每个按钮都一样重。

### P2

- 为后续 Playwright、Lighthouse、axe-core 介入预留稳定页面结构。
- 为后续视觉回归测试保留稳定的页面标识和结构分区。

## 5. Design

### Overall Approach

- 采用“专业交易终端”视觉语言：深色背景、高对比文字、紧凑状态条、少量强调色。
- 采用“决策优先”信息结构：左侧负责解释与动作，右侧负责图表与上下文。
- 所有核心页面都遵循同一骨架：
  - 顶部摘要条
  - 双栏主区
  - 下方次要信息区

### Key Decisions

- 不做传统后台式长页面，避免信息一路往下堆。
- 不做纯交易台布局，避免把研究结果压缩成附属信息。
- 首页不再只是总览，而是全局入口和下一步动作面板。

### Alternatives

#### 方案 A：仪表板式总览

- 优点：容易做，改动小。
- 缺点：专业终端感不足，图表和执行被弱化。

#### 方案 B：双栏交易研究终端

- 优点：最符合当前 `Qlib + Freqtrade + 控制平面` 结构。
- 缺点：需要统一多个页面布局。

#### 方案 C：传统多面板交易台

- 优点：交易终端味最强。
- 缺点：研究链会被压扁，容易继续拥挤。

### Impact Scope

- `apps/web/app/page.tsx`
- `apps/web/app/signals/page.tsx`
- `apps/web/app/market/[symbol]/page.tsx`
- `apps/web/app/strategies/page.tsx`
- `apps/web/components/app-shell.tsx`
- `apps/web/components/*` 中与页面骨架、状态条、候选板、图表区直接相关的组件
- `apps/web/app/globals.css`

## 6. Acceptance Criteria

- 首页、信号页、单币页、策略页都改成统一终端骨架。
- 页面主区不再出现“多个纵向大块一路堆到底”的主结构。
- 信号页和单币页能同时展示研究结果与下一步动作。
- 策略页能更快看见推荐执行候选和执行器状态。
- 真实页面 HTML 中能看到新的摘要条、双栏区和关键信息。

## 7. Validation Strategy

- 运行前端相关单元测试。
- 运行前端构建。
- 启动 `9011` 和 `9012` 做真实页面验证。
- 至少验证：
  - `/`
  - `/signals`
  - `/market/BTCUSDT`
  - `/strategies`

## 8. Risks & Rollback

### Risks

- 页面骨架改动较大，容易把现有数据块打散。
- 若层级处理不好，可能从“拥挤”变成“过空”。
- 开发服务器和生产构建产物可能出现本地不一致。

### Rollback

- 以页面为单位回退。
- 先保留现有 API 契约和组件数据输入，不在本轮改接口。
