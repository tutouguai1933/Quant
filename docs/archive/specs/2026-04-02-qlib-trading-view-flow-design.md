# Qlib 多周期交易视图与页面动线重构设计

## 背景判断

当前市场页、单币页、策略页已经接上统一研究字段，但整体体验还有两个明显问题：

- 图表层太弱，单币页还不具备真正的“看盘”感觉
- 页面职责还没有完全拉开，市场页、单币页、策略页之间的动线不够顺

用户现在最直接的痛点是：

- 需要像 Binance 一样看到更完整的周期切换
- 需要更直观的 K 线主图，而不是主要靠文字块理解当前判断
- 需要更清楚的页面路径，从“先看什么”到“要不要执行”一条线走通

所以这一步不再只是补几个字段，而是把图表层和页面动线一起重构。

## 总体结论

推荐方案固定为：

- **交易视图型：交易所 K 线做主图，Qlib 负责研究判断、信号打点、多周期摘要和解释**

也就是：

- 图表主数据继续来自交易所 K 线
- `Qlib` 不直接替代 K 线数据源
- `Qlib` 负责：
  - 研究信号
  - 趋势判断
  - 多周期摘要
  - 图上打点
  - 解释文本
- 页面动线改成：
  - 市场页先筛选
  - 单币页先看图再判断
  - 策略页最后做执行决策

## 核心设计目标

这一步只解决 5 件事：

- 单币页提供更完整的周期切换
- K 线主图成为单币页主角
- `Qlib` 研究结果真正变成图表上的辅助层
- 市场页、单币页、策略页职责彻底分开
- 三个页面仍保持同一套研究语言

## 页面职责重构

### 1. 市场页

市场页只负责“先筛选目标”，不负责详细解释。

这一页要回答：

- 哪些币值得先点进去
- 当前更偏突破还是回调
- 哪些币的判断更强
- 哪些币存在研究分歧，需要优先看图

这一页保留表格，但会更像筛选入口，重点字段包括：

- Symbol
- 最新价
- 24h 变化
- 多周期状态
- 研究倾向
- 推荐策略
- 判断信心
- 主判断

同时补一个更直观的顶部摘要区：

- 突破候选数
- 回调候选数
- 高信心数量
- 当前最值得看图的 symbol

### 2. 单币页

单币页是整条链路的主战场。

这一页负责回答：

- 当前图上发生了什么
- 现在应该看哪个周期
- 图上的关键信号点、入场位、止损位在哪里
- `Qlib` 研究层是在支持、压低还是观望
- 现在下一步该做什么

页面结构固定成：

- 顶部控制条
  - Symbol
  - 交易所来源
  - 周期切换
  - 指标开关
  - 研究打点开关
- 主图表区
  - K 线
  - 当前价格
  - 入场线
  - 止损线
  - 信号点
- 右侧固定判断卡
  - 推荐策略
  - 研究倾向
  - 判断信心
  - 研究门控
  - 主判断
  - 入场参考 / 止损参考
- 主图下方辅助区
  - 多周期摘要
  - 研究解释
  - 下一步动作

### 3. 策略页

策略页只负责“能不能执行”，不再重复承担图表解释工作。

这一页保留：

- 两套策略卡片
- 执行器状态
- 最近信号
- 最近订单 / 持仓
- 执行动作

每张策略卡继续保留统一研究字段：

- 推荐策略
- 研究倾向
- 判断信心
- 研究门控
- 主判断
- 研究解释

但不再尝试承担“看图”角色。

## 图表主数据与 Qlib 分工

这是这一轮最重要的边界。

### 图表主数据

图表主数据继续使用交易所 K 线。

原因：

- 周期切换应该直接对齐 Binance 习惯
- K 线是交易终端的主信息源
- 交易所数据更适合承担真实看盘职责

### Qlib 负责内容

`Qlib` 不直接替代 K 线，而是负责图表上的研究层：

- 当前周期研究倾向
- 多周期研究摘要
- 推荐策略判断
- 研究门控状态
- 信号点
- 入场参考
- 止损参考
- 研究解释

可以理解为：

- 交易所 K 线是“事实层”
- `Qlib` 是“解释层”

## 周期设计

时间周期默认按 Binance 常见习惯对齐，第一轮支持：

- `1m`
- `3m`
- `5m`
- `15m`
- `30m`
- `1h`
- `4h`
- `1d`
- `1w`

说明：

- `1m / 3m / 5m / 15m / 30m` 适合更细的短线观察
- `1h / 4h` 适合当前两套波段策略主判断
- `1d / 1w` 适合更上层趋势确认

多周期摘要第一轮不需要覆盖所有可切换周期，而是固定展示最关键的 4 层：

- `1d`
- `4h`
- `1h`
- `15m`

## 推荐方案与备选方案

### 方案 1：交易视图型

这是推荐方案。

特点：

- K 线做主角
- `Qlib` 做解释层
- 单币页成为主战场
- 三页职责最清晰

优点：

- 最符合用户当前直觉
- 最容易承接 Binance 使用习惯
- 最适合后续继续加更多图表层功能

问题：

- 需要同时改前端结构和部分市场接口
- 范围明显大于上一轮轻量图层

### 方案 2：研究主导型

不推荐。

特点：

- 研究判断放在最前
- 图表只是解释背景

问题：

- 不够像交易终端
- 不能直接解决“看盘不直观”的核心痛点

### 方案 3：双层驾驶舱型

当前阶段不推荐。

特点：

- 页面会更像控制面板
- 信息密度更高

问题：

- 图表反而不突出
- 容易把单币页做得过重

## 数据与接口方向

这一步尽量在现有接口基础上扩，不做无边界重写。

预计需要的方向包括：

- 市场页：
  - 在现有 `research_brief` 基础上补多周期摘要所需字段
- 单币页：
  - `GET /api/v1/market/{symbol}/chart` 需要支持更多周期参数
  - 继续返回标准化 K 线
  - 补多周期研究摘要
  - 保留 `research_cockpit`
  - 保留 `markers`
- 策略页：
  - 继续复用现有 `research_cockpit`

关键原则：

- 前端不自己推导研究结论
- 后端统一给出研究和图表需要的摘要结构
- 页面只负责渲染和交互

## 降级规则

这一步继续保持明确降级，不允许静默失败。

### 1. 某个周期没有图表数据

- 页面明确显示该周期暂无图表
- 不让整个单币页一起空掉

### 2. 某个周期没有研究结果

- K 线仍可显示
- 研究区明确显示“该周期暂无研究结论”

### 3. 标记数据缺失

- 图表仍正常显示
- 入场线 / 止损线 / 信号点单独缺省

### 4. 市场 API 短暂失败

- 市场页继续显示空状态兜底
- 但要避免误导用户，把文案调整得更清楚

## 实施边界

这次要做：

- 多周期图表切换
- 单币页主战场重构
- 市场页筛选入口重构
- 策略页执行页收口
- `Qlib` 多周期摘要和图表辅助层接入
- 测试、review、真实页面验证、文档和提交

这次不做：

- 新增独立研究页面
- 直接接完整图表库之外的重型交易终端功能
- 自动下单
- 改软门控规则
- 改 `Freqtrade` 执行边界

## 验证原则

至少覆盖：

- 市场页能显示新的筛选字段和优先关注逻辑
- 单币页支持多周期切换
- 单币页图表层会显示研究打点和风险位
- 策略页只保留执行相关视图
- API 测试通过
- 前端测试通过
- 类型检查和构建通过
- 真实页面验证通过

## 对多 session 协同的影响

这一步仍然主要属于 `Qlib` 研究展示层和市场/图表体验收口，不改执行边界。

所以：

- 我这边继续负责图表层、研究层和页面动线
- 另一个 session 继续负责 `Freqtrade` 执行联调
- 双方仍通过统一研究字段和执行状态契约衔接
