# Quant 系统补强实施计划

这份计划的目标不是推翻现有系统，而是在已经跑通的 `Qlib + Freqtrade + Control Plane` 主链上，补齐个人量化系统真正长期可用的几层能力。

关联背景：

- 当前系统已经完成研究、筛选、`dry-run`、小额 `live`、复盘的最小闭环
- 当前最缺的不是新页面，而是数据、分析、回测、评估、执行稳定性和自动化可靠性

## 设计参考原则

后续补强不是“往系统里塞更多指标”，而是按适合个人开发者的节奏，把指标和因子收成少而精、可解释、可回测、可复盘的结构。

当前计划明确参考这些原则：

- 指标按用途分层：
  - 趋势类
  - 动量类
  - 震荡类
  - 成交量类
  - 波动率类
- 量化因子优先保持少而精，不追求一开始堆很多
- 指标和因子要组合使用，不依赖单一信号
- 指标参数要按交易周期调整，当前主目标仍是 `1-3 天` 单币择时
- 要先判断行情状态，再决定该用哪类指标：
  - 趋势行情优先趋势类和动量类
  - 震荡行情优先震荡类和量价确认
- 不迷信指标本身，最终必须经过回测、验证和真实执行复盘
- 风控仍然内置，不允许指标信号绕过仓位、白名单、金额上限和停机规则

## 范围

- 补强数据层
- 补强分析 / 因子层
- 升级回测层
- 独立评估与复盘层
- 深化研究层
- 工程化执行层
- 系统化自动化层

## 不在本轮范围

- 多市场扩展
- 高频策略
- `Lean / vn.py` 主链实现
- 大规模参数搜索
- 贝叶斯优化平台化
- 全自动无人值守实盘

- 让研究、回测、执行使用的数据口径更统一

- [x] 明确 `raw / cleaned / feature-ready` 三种数据状态
- [x] 补一份数据状态说明文档
- [x] 明确当前研究层读取的是哪一层数据

### Task 1.2：数据快照标准化

- [x] 统一训练、推理、回测的数据快照命名
- [x] 让研究报告和复盘都能回指具体快照
- [x] 明确数据快照和实验账本的关联方式

### Task 1.3：数据缓存与复用

- [x] 避免重复拉取同一批 K 线
- [x] 让研究和回测优先复用已有缓存
- [x] 明确缓存失效和刷新规则

验收：

- [x] 任意一轮训练都能说清“用了哪份数据”
- [x] 训练、推理、回测在日志和报告里能对上同一批快照

## Phase 2：Analysis / 因子层独立

目标：

- 不再把因子逻辑散在研究主流程里
- 让特征工程能单独维护、单独对比、单独扩展

### Task 2.1：因子分类

- [x] 按趋势、动量、波动、量能四类整理已有因子
- [x] 补充震荡类观察指标，并明确哪些只用于辅助、不直接入主模型
- [x] 为每类因子补一句用途说明
- [x] 明确哪些是基础因子，哪些是组合因子

当前优先方向：

- 趋势类：均线、均线偏离、趋势位置
- 动量类：收益变化、突破强度、ROC 一类
- 震荡类：RSI / KDJ / CCI 这类超买超卖参考
- 成交量类：成交量相对均值、量价确认
- 波动率类：ATR、波动率区间、突破环境

### Task 2.2：统一预处理规则

- [x] 明确标准化规则
- [x] 明确缺失值处理规则
- [x] 明确去极值规则
- [x] 明确哪些因子需要做周期参数映射，避免短线参数直接套到 `1-3 天` 持有周期

### Task 2.3：因子输出协议

- [x] 统一因子层输入输出格式
- [x] 让研究训练和推理都走同一因子出口
- [x] 为因子层补最小回归测试
- [x] 明确“主判断因子”和“辅助确认因子”的字段区分

验收：

- [x] 能单独列出当前系统到底用了哪些因子
- [x] 训练和推理不再各自拼一套特征逻辑

## 推荐执行顺序

按当前系统状态，后续开发建议严格按下面顺序推进：

1. 数据层补强
2. Analysis / 因子层独立
3. 回测层升级
4. 评估与复盘层独立
5. 研究层深化
6. 执行层工程化
7. 自动化系统化

```bash

cd /home/djy/Quant
python -m unittest discover -s services/api/tests -v
python -m unittest discover -s services/worker/tests -v
python -m unittest discover -s tests -v
```

```bash
cd /home/djy/Quant/apps/web

```
