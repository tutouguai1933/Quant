# I + J 视觉统一与多因子主线补强设计

## 1. 目标

这次只做一件事：

- 把前端已经完成的“摘要优先、细节按需展开”继续收口成统一体验
- 同时把 `因子挖掘 -> 因子验证 -> 去冗余 -> 候选篮子 -> 执行篮子` 这条主线明确显化出来

范围按已经确认的口径执行：

- 一起做 `I /视觉和交互统一` 与 `J /多因子主线补强`
- 以 `/features -> /evaluation -> /strategies` 为主线
- 首页只补最短入口，不把首页做成第二个因子页
- 前端术语直接统一成 `候选篮子 / 执行篮子`

## 2. 这次不做什么

- 不改后端接口字段名
- 不重写研究、评估、执行逻辑
- 不新增新的一级页面
- 不做整站换肤式视觉重做
- 不把首页扩展成完整因子分析页

## 3. 当前问题

虽然前端的主工作台、研究页、评估页、执行页、任务页已经完成了“摘要优先”的第一轮重构，但现在还剩两类明显问题。

第一类是体验口径还没有完全统一：

- 不同页面的摘要卡层级还不完全一致
- 抽屉的描述、按钮和内容密度还不完全一致
- 空态、错误态、加载态的说法还不够统一
- 某些页面仍然带着“开发面板式”的细节残留

第二类是多因子主线还没有真正显化出来：

- `/features` 已经是“因子工作台”，但更像“5 张因子摘要卡”
- `/evaluation` 和 `/strategies` 仍然大量使用“候选池 / live 子集”这类旧口径
- 用户不容易一眼看懂：因子页的分析结果，最后是怎样流向评估和执行的

## 4. 术语决定

本次前端统一采用以下术语：

- `候选池` -> `候选篮子`
- `live 子集` -> `执行篮子`

说明：

- 后端字段和接口数据先不改，避免放大改动范围
- 前端只在展示层做映射
- 文案要保证非技术用户也能读懂

语义约束：

- `候选篮子`：研究和评估阶段当前保留、允许继续比较的一组标的
- `执行篮子`：从候选篮子里继续经过更严门控，允许进入 `dry-run / live` 的更小一组标的

## 5. 总体方案

采用“统一组件层 + 主线文案层 + 跨页承接层”三层做法。

### 5.1 统一组件层

目标是把视觉和交互的骨架真正统一。

具体做法：

- 继续以 `SummaryCard`、`SectionShell`、`DetailDrawer` 为核心
- 把摘要卡统一成同一节奏：
  - `eyebrow`
  - `title`
  - `summary`
  - `detail`
  - `status / actions / footer`
- 把抽屉统一成同一节奏：
  - 标题
  - 一句用途说明
  - 2 到 4 个重点摘要块
  - 再下沉到表格、配置或说明

这层重点不是“更花”，而是“更稳、更一致”。

### 5.2 主线文案层

目标是让多因子流程能被直接读懂。

具体做法：

- `/features` 顶部新增一条明确的流程表达
- 这条表达固定用五步：
  - 因子挖掘
  - 因子验证
  - 去冗余
  - 候选篮子
  - 执行篮子
- 每一步都要有一句“现在到哪了”的说明
- 当前已有的因子分类、有效性、冗余、总分解释，都挂到这条主线上

### 5.3 跨页承接层

目标是让用户知道这条流程不是停在因子页。

具体做法：

- `/features` 明确告诉用户：因子页的结果会流向候选篮子和执行篮子
- `/evaluation` 明确展示：候选篮子里现在是谁、为什么留下、为什么淘汰
- `/strategies` 明确展示：执行篮子从哪里来、当前放行了谁、为什么只剩这几个
- 首页只保留一个最短入口：告诉用户当前主线走到哪一步，以及下一步该去哪里

## 6. 页面设计

## 6.1 `/features`

这是这次改动的核心页。

页面顶部新增一条“多因子主线带”，用于显化：

- 因子挖掘：当前启用了哪些因子类型
- 因子验证：哪些因子当前更有效
- 去冗余：哪些同类因子需要合并或降级
- 候选篮子：当前哪些结论会进入后续比较
- 执行篮子：最终哪些标的能继续进入执行链

现有 5 张摘要卡继续保留，但职责重新挂接：

- `因子分类总览` -> 因子挖掘
- `当前启用因子` -> 因子挖掘 / 当前协议
- `因子有效性摘要` -> 因子验证
- `因子冗余摘要` -> 去冗余
- `总分解释入口` -> 候选篮子形成逻辑

同时补两类新承接信息：

- 一张轻量级“候选篮子去向说明”
- 一张轻量级“执行篮子去向说明”

这两块不做成新的复杂主卡，而是作为主线带里的后两步说明，避免页面重新变长。

## 6.2 `/evaluation`

评估页负责把“候选篮子”讲清楚。

要改的重点：

- 首屏文案从“候选池”统一切到“候选篮子”
- 推荐摘要、淘汰摘要、门控详情统一围绕“候选篮子”表达
- 抽屉里把这几件事说清楚：
  - 当前候选篮子里有哪些标的
  - 为什么它们还留在篮子里
  - 为什么有些标的没有进入候选篮子
  - 执行篮子是从候选篮子里怎样继续缩出来的

评估页的职责不变，但表达更像“分层筛选过程”，而不是“孤立判断页”。

## 6.3 `/strategies`

执行页负责把“执行篮子”讲清楚。

要改的重点：

- 把“候选池详情”统一改成“候选篮子详情”
- 把 `live 子集` 统一改成“执行篮子”
- 在执行摘要里明确表达：
  - 候选篮子是上游输入
  - 执行篮子是当前允许进入执行链的更小子集
  - 执行器只对执行篮子负责，不对整组候选篮子负责

用户应该能在策略页一眼看懂：

- 上游给了多少
- 当前放行了多少
- 现在真正允许执行的是谁

## 6.4 首页 `/`

首页只补最短入口，不扩页。

只做两件事：

- 在首页工作台里用一句更直白的话说明当前主线走到哪一步
- 给出一个稳定入口，把用户送到 `/features`、`/evaluation` 或 `/strategies`

首页不新增新的大区块，不重复因子页和评估页的信息。

## 7. 统一体验规则

## 7.1 摘要卡

统一规则：

- 标题只说“当前是什么”
- `summary` 只说“当前结论”
- `detail` 只说“为什么值得看”
- 详情抽屉只负责“展开证据”，不承担第二主动作区

## 7.2 抽屉

统一规则：

- 抽屉先给一句用途说明
- 再给 2 到 4 个重点摘要块
- 最后再给长说明、表格或配置
- 抽屉内按钮命名统一成“查看…详情 / 打开…弹窗 / 去…页”

## 7.3 反馈态

统一规则：

- 加载态：告诉用户“现在正在准备什么”
- 空态：告诉用户“当前为什么为空，下一步该去哪”
- 错误态：告诉用户“当前哪层不可用，先回哪一页处理”

不再使用一批页面一个说法。

## 8. 技术实现边界

这次优先改前端展示层。

主要涉及：

- `apps/web/components/summary-card.tsx`
- `apps/web/components/detail-drawer.tsx`
- `apps/web/components/features-*`
- `apps/web/components/evaluation-*`
- `apps/web/components/strategies-*`
- `apps/web/app/features/page.tsx`
- `apps/web/app/evaluation/page.tsx`
- `apps/web/app/strategies/page.tsx`
- `apps/web/app/page.tsx`

可以接受的辅助改动：

- 补统一的展示辅助组件
- 补文案映射函数
- 补前端测试

这次不改：

- 后端接口协议
- 数据库结构
- 研究执行算法

## 9. 测试与验证

先补失败测试，再实现。

重点验证 4 件事：

1. `/features` 首屏能直接看出五步主线  
2. `/evaluation` 首屏和抽屉能直接看到“候选篮子”表达  
3. `/strategies` 首屏和抽屉能直接看到“执行篮子”表达  
4. 首页能把用户送到正确主线页面，而不是重复讲整套流程

验证方式：

- 定向 Playwright 页面回归
- 前端构建
- 真实页面联调

## 10. 风险

主要风险有 3 个：

- 前端文案改动面较广，旧测试容易一起受影响
- 如果首页加得过多，会重新变成第二个信息堆叠页
- 如果 `/features` 新增块过多，可能反而破坏已经完成的“摘要优先”

对应约束：

- 首页只补最短入口
- 因子页新增主线带，但不新增大面积复杂表格
- 旧术语只在数据层保留，不在页面标题和主文案里继续出现

## 11. 结论

这次不是再加新页面，而是把已经存在的因子、评估、执行三页真正串成一条能看懂的主线。

完成后，用户应该能直接回答这 3 个问题：

- 当前因子体系在做什么
- 当前哪些标的已经进入候选篮子
- 当前哪些标的已经进入执行篮子
