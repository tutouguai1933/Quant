# Quant 前端工作台重构执行计划

这份计划是 [docs/frontend-workbench-redesign.md](/home/djy/Quant/docs/frontend-workbench-redesign.md) 的落地版。

目标只有一个：

- 把当前“信息过多、默认全展开”的前端，重构成“摘要优先、细节按需展开”的工作台

## 使用方式

- 默认按顺序执行
- 每做完一个大阶段，都要：
  - 更新 `CONTEXT.md`
  - 做 code review
  - 做定向测试
  - 做真实页面联调
  - 再进入下一阶段

## 总体顺序

1. 先重构信息架构基础
2. 再做主工作台
3. 再做因子工作台
4. 再做研究页收窄
5. 再做决策页收窄
6. 再做执行 / 运维页收窄
7. 最后做整体验收

## A. 信息架构基础

- [x] A1. 固定一级工作区和二级工具区
  - 目标：主导航只保留主线页面，工具页降级
  - 重点文件：
    - `apps/web/components/app-shell.tsx`
  - 结果标准：
    - 主导航和工具入口分区
    - 用户一眼能看到主线入口

- [x] A2. 统一“摘要卡 + 详情抽屉 + 弹窗”三层结构
  - 目标：后续页面都用同一套信息层级
  - 重点文件：
    - `apps/web/components/`
  - 建议新增：
    - `summary-card`
    - `detail-drawer`
    - `detail-dialog`
    - `section-shell`

- [x] A3. 统一状态词
  - 目标：全站统一使用“正常 / 运行中 / 阻塞 / 需人工处理”
  - 重点文件：
    - `apps/web/lib/api.ts`
    - `apps/web/components/status-*`

- [x] A4. 统一主动作区规范
  - 目标：每个主页面只保留一个主动作区
  - 当前进度：`/tasks`、`/strategies`、`/evaluation`、`/`、`/research` 都已收口为单一主动作区
  - review 重点：
    - 页面里不能出现多个同级主按钮组

- [x] A5. 做 `A` 阶段 review / build / 页面联调
  - 测试重点：
    - 导航不乱
    - 页面不白屏
    - 抽屉 / 弹窗基础交互正常

## B. 主工作台 `/`

- [x] B1. 新建首页主工作台框架
  - 目标：用一个页面收主链 80% 的日常入口
  - 重点文件：
    - `apps/web/app/page.tsx`

- [x] B2. 新增“当前推荐”卡
  - 目标：默认只显示当前最该关注的候选和一句推荐摘要
  - 数据来源：
    - `evaluation`
    - `automation`

- [x] B3. 新增“当前研究状态”卡
  - 目标：让首页直接看到研究是否在跑、跑到哪
  - 数据来源：
    - `research runtime`

- [x] B4. 新增“当前执行状态”卡
  - 目标：让首页直接看到执行器是否可用、最近订单摘要
  - 数据来源：
    - `strategies`
    - `balances / orders / positions`

- [x] B5. 新增“当前风险与告警”卡
  - 目标：让首页直接看到头号阻塞
  - 数据来源：
    - `tasks / automation`

- [x] B6. 新增“当前下一步动作”卡
  - 目标：明确告诉用户现在该去哪一页、做什么
  - 数据来源：
    - `arbitration`

- [x] B7. 新增“最近结果回看”卡
  - 目标：显示最近一轮研究 / 执行 / 复盘摘要

- [x] B8. 给每张卡补“查看详情”抽屉
  - 目标：默认不铺细节，但允许快速深入

- [x] B9. 做 `B` 阶段 review / test / 页面联调
  - 测试重点：
    - 首页能覆盖日常主链入口
    - 不登录时也能看到安全摘要
    - 登录后动作反馈正常

## C. 因子工作台 `/features`

- [x] C1. 重新定义 `/features` 为“因子工作台”
  - 目标：不再只是权重配置页

- [x] C2. 新增“因子分类总览”
  - 默认显示：
    - 趋势
    - 动量
    - 震荡
    - 成交量
    - 波动率
    - 扩展因子

- [x] C3. 新增“当前启用因子”卡
  - 目标：明确当前主判断因子、辅助因子和权重

- [x] C4. 新增“因子有效性摘要”卡
  - 目标：显化因子 IC、分组收益、稳定性
  - 后端依赖：
    - 如果当前缺摘要接口，需要补最小接口

- [x] C5. 新增“因子冗余摘要”卡
  - 目标：显化高相关因子和去重状态
  - 后端依赖：
    - 如果当前缺相关矩阵摘要，需要补最小接口

- [x] C6. 新增“总分解释入口”
  - 目标：能解释当前候选为什么拿到这个总分

- [x] C7. 新增因子详情抽屉
  - 抽屉内容：
    - 因子说明
    - 时间序列
    - IC
    - 分组收益
    - 稳定性
    - 相关性

- [x] C8. 做 `C` 阶段 review / test / 页面联调
  - 测试重点：
    - 用户能看懂因子体系
    - 默认视图不再堆满图表和表格

## D. 研究工作台 `/research`

- [x] D1. 把默认视图收窄为“当前状态 + 当前配置摘要 + 当前产物”

- [x] D2. 把完整配置区移入“配置抽屉”

- [x] D3. 把模板 / 模型 / 标签说明移入可展开详情

- [x] D4. 把实验对比移入“实验弹窗”
  - 目标：研究页不承担完整实验中心职责

- [x] D5. 保留研究运行状态为页面核心区块
  - 目标：触发动作后能立刻看到状态变化

- [x] D6. 做 `D` 阶段 review / test / 页面联调

## 第一批推荐执行顺序

优先按这个顺序：

1. A 信息架构基础
2. B 主工作台
3. E 决策工作台
4. G 运维工作台
5. F 执行工作台
6. D 研究工作台
7. C 因子工作台
8. H 工具页收口
9. I 交互统一
10. J 多因子主线补强
11. K 综合验收

## 为什么是这个顺序

- 先改外壳和首页，立刻降低信息负担
- 再收决策、运维、执行三页，因为这三页现在最拥挤
- 再回头收研究页和因子页
- 最后再做因子主线深化
