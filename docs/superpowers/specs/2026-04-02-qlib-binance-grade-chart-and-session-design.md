# Qlib 专业 K 线终端、登录保持与市场页提速设计

## 背景判断

当前 `Qlib` 这条线已经完成：

- 市场页先筛选
- 单币页先看图再判断
- 策略页最后做执行决策
- `1m / 3m / 5m / 15m / 30m / 1h / 4h / 1d / 1w` 周期切换
- 单币页研究侧卡、多周期摘要和最小图表打点

但用户当前的 4 个痛点仍然非常明确：

1. 当前 K 线主图还是静态 SVG，不具备专业交易终端应有的交互能力
2. 登录状态不能稳定保持，重复登录很烦，而且存在点击登录后卡住的体验问题
3. 市场页进入速度慢，首屏等待感明显
4. 周期按钮位置还不够顺手，需要更贴近 Binance 的操作习惯

这一步不再只是“继续补图层”，而是把单币页升级成真正的专业交易主战场，并同时把登录保持和市场页速度一起收口。

## 总体结论

推荐方案固定为：

- **专业金融图表库接管主图，Qlib 继续负责研究判断、信号打点、多周期摘要和解释**

也就是：

- 图表主数据继续来自交易所 K 线
- 主图渲染改成专业 K 线图表库，而不是继续自绘 SVG
- `Qlib` 继续负责：
  - 研究倾向
  - 推荐策略
  - 研究门控
  - 入场参考 / 止损参考
  - 多周期摘要
  - 研究解释
  - 图上信号点和研究标记
- 页面动线保持不变：
  - 市场页先筛选
  - 单币页先看图再判断
  - 策略页最后做执行决策
- 登录体验升级为：
  - 默认保持 7 天
  - 进入页面不再频繁重复登录
  - 登录成功不再出现卡住感

## 核心设计目标

这一步只解决 4 件事：

1. 单币页主图达到接近 Binance 的专业交互体验
2. 登录状态稳定保持 7 天，明显减少重复登录
3. 市场页做到“先出壳，再补数据”，显著降低等待感
4. 周期切换位置和交互习惯更贴近交易终端

## 单币页重构

### 1. 页面定位

单币页继续作为整条链路的主战场，但这次图表真正成为主角。

这一页负责回答：

- 当前图上发生了什么
- 现在应该看哪个周期
- 关键入场位、止损位、信号点在哪里
- `Qlib` 研究层是在支持、压低还是观望
- 现在下一步该做什么

### 2. 页面结构

单币页固定成三段：

- 图表左侧或右侧的快速周期条
- 中间主图区域
- 右侧固定判断与动作卡

详细结构如下：

- 周期快速切换条
  - `1m / 3m / 5m / 15m / 30m / 1h / 4h / 1d / 1w`
  - 第一轮优先放在主图左侧
  - 预留在右侧增加二级切换位的空间
- 主图区域
  - 专业 K 线主图
  - 横纵坐标
  - 十字光标
  - 鼠标拖拽平移
  - 滚轮缩放
  - 成交量副图
  - 入场线
  - 止损线
  - 信号点
- 右侧固定判断卡
  - 推荐策略
  - 研究倾向
  - 判断信心
  - 研究门控
  - 主判断
  - 入场参考 / 止损参考
  - 执行动作入口
- 主图下方
  - 多周期摘要
  - 研究解释
  - 下一步动作说明

### 3. 图表交互范围

这一轮最低必须包含：

- 鼠标拖拽平移
- 滚轮缩放
- 横纵坐标与价格刻度
- 十字光标和悬浮信息
- 多周期切换时只重绘图表，不整页刷新
- 成交量副图

这一轮暂不包含：

- 高级画线工具
- 深度图 / 订单簿
- 自定义指标编辑器

## 图表主数据与 Qlib 分工

这是这一轮最重要的边界。

### 图表主数据

图表主数据继续使用交易所 K 线。

原因：

- 周期习惯直接对齐 Binance
- K 线是交易终端的事实层
- 真实看盘不能让研究层替代行情层

### Qlib 负责内容

`Qlib` 不替代 K 线，而是继续承担解释层：

- 当前周期研究倾向
- 多周期研究摘要
- 推荐策略判断
- 研究门控状态
- 入场参考
- 止损参考
- 研究解释
- 图上信号点和研究打点

可以理解为：

- 交易所 K 线是“事实层”
- `Qlib` 是“解释层”

## 登录与会话设计

### 1. 登录保持

登录状态默认保持：

- **7 天**

在这 7 天内：

- 不要求用户每次进入都重新登录
- 用户只在以下情况重新登录：
  - 主动退出
  - token 真实失效
  - 后端明确拒绝当前会话

### 2. 登录体验

当前登录的两个问题是：

- 登录成功后仍可能反复看到未登录状态
- 点击登录后存在卡住感

这一步的设计要求是：

- 登录成功后直接进入目标页
- 页面外壳不要因为等待完整会话校验而卡住
- 真正受保护的动作继续由服务端做最终校验
- 会话失效时优先做轻提示和局部引导，而不是整页卡死

### 3. 会话分层

这一步会把会话体验拆成两层：

- 页面可见性层：
  - 优先保证页面快速可见
- 动作保护层：
  - 启动策略 / 派发信号 / 研究训练 / 研究推理等动作继续严格校验

这样既保证安全，也不会让“看页面”每次都像重新登录。

## 市场页性能设计

### 1. 当前问题

市场页当前是服务端进页就等待完整市场数据，用户感受会变成：

- 点击市场
- 等待
- 整页一起出来

这和专业交易终端的使用预期不一致。

### 2. 目标体验

这一步固定目标为：

- 进入市场页先秒开骨架
- 表格数据在后台补进来
- 返回市场页优先复用最近结果
- 尽量避免每次都整页重新等待

### 3. 数据策略

市场页改成：

- 页面先输出稳定骨架
- 表格数据走客户端拉取
- 客户端保留短时缓存
- 切到单币页再切回时，优先复用最近一次市场结果

这一步追求的是体感速度，而不是去做复杂缓存系统。

## 周期按钮位置设计

用户明确要求：

- `1d / 1m` 这些按钮尽量放在图的左右两边，更方便切换

推荐第一轮做法：

- 把主周期切换条固定在主图左侧
- 右侧预留放次级图层操作位

原因：

- 左侧竖排更节省顶部空间
- 更接近交易终端对主图区域的重视方式
- 能把主图顶部留给价格、状态、悬浮信息

如果实际落地后发现左侧过长或移动端压缩严重，再考虑：

- 桌面端左侧竖排
- 移动端顶部横排

## 推荐方案与备选方案

### 方案 1：继续增强自绘 SVG

不推荐。

优点：

- 依赖更少
- 保持当前实现路径

问题：

- 很难真正接近 Binance 级别交互
- 后续越补越重，返工概率高
- 坐标、拖拽、缩放、十字光标、成交量副图都要自己重新造轮子

### 方案 2：专业图表库接管主图

这是推荐方案。

优点：

- 最接近用户的目标体验
- 交互能力成熟
- 我们只需要把研究层、周期切换和动作区接上
- 长期维护成本更低

问题：

- 需要增加前端依赖
- 需要重做单币页图表组件边界

### 方案 3：整页控制台大重构

当前阶段不推荐。

优点：

- 可以一步把市场页、单币页、策略页做成完整终端壳

问题：

- 范围太大
- 会把“图表、登录、性能”这三个关键痛点混在一起
- 本轮更容易失控

## 数据与接口方向

这一步尽量在现有接口基础上扩，不做无边界重写。

预计方向：

- 市场页：
  - 保持 `GET /api/v1/market`
  - 前端改成客户端拉取和短缓存
- 单币页：
  - 保持 `GET /api/v1/market/{symbol}/chart`
  - 继续使用：
    - `active_interval`
    - `supported_intervals`
    - `multi_timeframe_summary`
    - `markers`
    - `research_cockpit`
    - `strategy_context`
- 会话：
  - 登录写入更稳定的长期 cookie
  - 会话读取逻辑改成更适合快速页面渲染

## 验证要求

这一步完成后必须至少验证：

### 自动测试

- 前端测试补到：
  - 图表组件不是静态 SVG 主实现
  - 周期切换不走整页刷新
  - 登录保持链路存在
  - 市场页改成客户端取数和骨架加载
- 现有 API / worker / frontend 测试继续通过
- `tsc` 和 `build` 继续通过

### 真实页面验证

必须真实验证：

1. 单币页图表支持：
   - 拖拽平移
   - 滚轮缩放
   - 横纵坐标
   - 十字光标
   - 成交量副图
2. 周期按钮已经贴近主图，并能顺畅切换
3. 登录成功后刷新页面仍保持登录状态
4. 登录动作不再卡住
5. 市场页进入时先出骨架，再补表格
6. 返回市场页时明显比现在更快

## 当前边界

这一步明确不做：

- 深度图
- 订单簿
- 高级画线工具
- 完整技术指标编辑器
- 完整实验平台
- Freqtrade 执行链路重构

## 实现顺序建议

推荐顺序固定为：

1. 先做单币页图表内核替换
2. 再做周期切换与研究打点接线
3. 再做登录保持与登录卡住修复
4. 再做市场页客户端取数与骨架加载
5. 最后统一做测试、真实验证和收尾文档

这个顺序的原因是：

- 图表是用户最直接的主诉
- 登录和市场页速度问题都和页面加载策略有关，放在图表边界稳定后统一处理更稳
