# Quant 平台架构回归设计

## 背景判断

项目现在没有真的“走错路”，而是因为前一阶段把 `crypto + Binance + dry-run` 做得更实，导致原始总设想里的几层暂时没有被明确保留下来。

当前已经做实的是：

- 控制平面基础骨架
- Binance 真实行情与余额
- Freqtrade 的最小执行适配边界
- 两套首批波段策略
- WebUI 的最小工作台

当前暂时被弱化的是：

- `Qlib` 研究层
- `Lean` 美股扩展位
- `vn.py` 中国市场扩展位
- `OpenClaw` 运维自动化层

所以接下来不应该推倒重来，也不应该自己重做一套执行引擎或研究平台，而是把总架构重新立起来，并在这个架构下继续把当前 `crypto + Binance` 做深。

## 总体结论

推荐路线固定为：

- **适配器优先，总控自研**

也就是：

- 我们自己做：`WebUI + Backend + Task Orchestrator + 标准契约 + 统一配置 + 审计`
- 不自己重做：`Freqtrade` 的加密执行能力、`Qlib` 的研究能力、`Lean` 的美股框架、`vn.py` 的中国市场框架
- `OpenClaw` 作为运维代理和自动化入口接入，不变成交易主链路

## 目标架构

### 1. 控制平面

组成：

- `WebUI`
- `Backend API`
- `Task Orchestrator`
- `Config / Secrets / Audit`
- `OpenClaw` 接入层

职责：

- 提供统一操作界面
- 管理任务、运行状态、风控结果、执行结果
- 编排研究、执行、同步、对账、归档等流程
- 统一权限、审计、告警和人工确认点

不负责：

- 不自己做交易撮合
- 不自己做完整研究框架
- 不直接实现多市场执行引擎

### 2. 研究层

核心组件：

- `Qlib`

职责：

- 数据准备
- 特征生成
- 训练
- 推理
- 输出信号、分数和解释
- 记录模型版本和实验信息

输出给控制平面的标准结果：

- `signal`
- `score`
- `reason / explanation`
- `model_version`
- `feature_snapshot`
- `run_id`

### 3. 执行层

按市场拆成多个执行适配器：

- `Freqtrade`：加密执行层
- `Lean`：未来美股 / 多市场扩展位
- `vn.py`：未来中国市场扩展位

职责：

- 接收控制平面的标准执行动作
- 管理策略实例、订单、成交、持仓、运行状态
- 返回各自市场的真实执行结果

第一阶段主链路只做：

- `Freqtrade`

### 4. 运维自动化层

核心组件：

- `OpenClaw`

职责：

- 触发训练
- 触发同步
- 触发对账
- 触发归档
- 触发巡检
- 触发告警处理

不负责：

- 不直接决定买卖
- 不绕过控制平面直接连交易所

## 为什么这样分

### 方案 1：适配器优先，总控自研

这是推荐方案。

优点：

- 保住原始总设想
- 不重复造轮子
- 后续扩市场时边界清楚
- 能继续沿着当前仓库推进，不需要推倒重来

缺点：

- 前期需要补更多接口边界和统一契约
- 文档和计划必须持续维护，否则容易再次“只剩一个局部工作台”

### 方案 2：全部自己做

不推荐。

问题：

- 研发量和风险都过大
- 会严重拖慢实盘能力落地
- 很容易演化成一个长期做不完的平台项目

### 方案 3：只做薄控制台，主要依赖外部系统

也不推荐。

问题：

- 体验会很碎
- 数据口径不统一
- 很难形成一个真正属于自己的个人操作空间

## 接下来 3 个阶段怎么走

### 阶段 1：把 Freqtrade 从“最小适配器”升级成真正的加密执行层

这是下一阶段的第一优先级。

要补的能力：

- 接入真实 Freqtrade REST API，而不是只用内存态模拟
- 显式管理 `demo / dry-run / live`
- 同步真实策略状态、订单、成交、持仓
- 支持控制平面的 `start / stop / pause`
- 把当前两套首批策略映射到 Freqtrade 可执行配置
- 明确“信号 -> 风控 -> 执行动作 -> Freqtrade” 的标准契约

完成后得到的结果：

- 当前 `crypto + Binance` 不再只是控制台演示，而是有真实执行器支撑的工作台

### 阶段 2：把 Qlib 从“最小占位流水线”升级成真正的研究层

这是下一阶段的第二优先级。

要补的能力：

- 明确 crypto 场景下的最小数据准备方式
- 把 K 线、成交量、基础指标转成研究输入
- 跑通最小训练、推理和 signal 输出
- 给策略页和图表页提供更可信的评分和解释
- 补模型版本、实验记录和运行记录

完成后得到的结果：

- 系统不再只是规则策略工作台，而是开始具备研究和 AI 辅助能力

### 阶段 3：保留多市场与运维自动化扩展位

这是结构层工作，不要求立刻做实。

要补的能力：

- 在目录、接口、契约、文档中明确 `Lean` 扩展位
- 在目录、接口、契约、文档中明确 `vn.py` 扩展位
- 在任务系统和 API 中明确 `OpenClaw` 的任务入口
- 保证以后接入时不需要推翻控制平面设计

完成后得到的结果：

- 项目重新对齐最初总设想
- 即使暂时只做 crypto，未来也不会再“看起来没有这些东西的影子”

## 每一层该自己做什么

### 我们自己做

- 控制平面 UI
- 控制平面 API
- 统一任务系统
- 统一配置与密钥管理
- 统一数据契约
- 统一审计与日志
- 统一风控与人工确认点
- 统一页面体验和工作流

### 不自己做

- 不自己做完整加密执行引擎
- 不自己做完整研究平台底座
- 不自己做美股执行框架
- 不自己做中国市场执行框架

## 多 agent / 多 session 的建议

这个项目适合多 agent，也适合多 session，但前提是主线必须单一。

推荐协作方式：

- 一个主 session 负责架构、拆任务、review、最终验证
- 多个子代理负责边界明确的小任务
- 不允许多个子代理同时改共享契约和同一核心文件

适合并行的工作：

- 前端页面
- 单个服务模块
- 单个适配器
- 单组测试
- 单份文档

不适合无约束并行的工作：

- 核心契约
- 全局配置模型
- 统一任务系统
- 订单 / 持仓 / 信号的标准结构

多 session 接力必须依赖这些文档：

- `CONTEXT.md`：当前进度
- `README.md`：功能和运行方式
- `docs/architecture.md`：当前架构
- `plan.md` 或阶段计划：下一步任务

## 当前优先级结论

接下来不应该把精力分散到 `Lean / vn.py` 的真实实现上。

正确顺序是：

1. 继续把当前 `crypto + Binance` 做深
2. 先把 `Freqtrade` 接成真正执行层
3. 再把 `Qlib` 接成真正研究层
4. 同时把 `Lean / vn.py / OpenClaw` 保留成明确的扩展位和任务位

## 下一步计划方向

基于这个设计，后续实现计划应拆成三条线：

### 线 1：Freqtrade 真实集成

- 真实 REST 适配器
- 策略实例控制
- 订单和持仓同步
- dry-run / live 边界

### 线 2：Qlib 最小研究层

- 数据准备
- 特征生成
- 最小训练
- 推理输出
- signal 解释

### 线 3：总架构回归

- `Lean` 扩展目录与接口
- `vn.py` 扩展目录与接口
- `OpenClaw` 任务入口与运维 API
- 文档和页面上的长期架构对齐

## 一句话总结

项目下一步不是推翻当前成果，而是在保留当前 `crypto` 主线的前提下，把 `Freqtrade`、`Qlib`、`Lean`、`vn.py`、`OpenClaw` 重新放回正确的位置：我们自己做总控，不自己重做执行和研究框架。
