# Quant 个人实盘量化工作台设计

> 状态：设计已收敛，待按阶段落地  
> 目标方向：选择“方案 B，按方案 A 的顺序落地”，最终演进到完整研究平台

## 1. 一句话定义

`Quant` 不再只是一个演示控制平面，而是要逐步演进成一个面向个人用户的加密量化实盘工作台。

这个工作台要能完成下面几件事：

- 接入真实市场数据
- 产出真实策略信号
- 展示趋势、仓位、订单、收益与风险
- 通过 Binance 真正买卖指定币种
- 支持从最小实盘闭环，逐步扩展到完整研究平台

## 2. 当前判断与范围收敛

### 2.1 用户目标

用户希望最终支持三类风格：

- 短线
- 波段
- 中长线

但当前阶段先做：

- `波段`

原因：

- 随机性比短线更低
- 更容易暴露系统 bug、执行 bug、策略 bug
- 对个人实盘更容易控制风险
- 更适合在项目早期快速验证真实交易闭环

### 2.2 当前阶段的关键设计决定

为了避免继续变成“大而全壳子”，当前先明确以下边界：

- 真实交易所先只接 `Binance`
- 真实交易品类先只做 `Binance Spot USDT 现货`
- 默认先不做合约，不做杠杆，不做多账户，不做跨交易所
- 真实实盘上线前，必须先经过：
  - 真实行情 + 模拟订单
  - 真实行情 + Freqtrade dry-run
  - 小资金实盘
- 研究平台最终会做，但不是当前第一批落地内容

这里的“先做 Spot 现货”是一个明确的收敛决策，不是最终上限。

## 3. 当前系统现状

当前仓库已经具备：

- 控制平面 API 骨架
- WebUI 最小驾驶舱
- 最小信号流水线
- 基础风控
- 内存态执行适配器
- 单管理员登录
- 演示级端到端流程

当前缺失的核心能力：

- 真实 Binance 市场数据接入
- 真实 Binance 账户 / 订单 / 持仓同步
- 真实币种池与可交易标的管理
- 趋势图、K 线图、信号标记、成交标记
- 可配置的波段策略中心
- 真实下单前后的风控和确认机制
- 交易日志、复盘、收益分析
- 真实 Freqtrade 集成，而不是内存假实现
- 策略参数、币种白名单、仓位规则的配置体系

## 4. 真实个人量化工作台必须具备的功能

下面按“必须有”“应该有”“后续再做”分层。

### 4.1 P0 必须有

这些功能缺任何一块，都还不能叫“真实个人量化工作台”。

#### A. 真实市场数据

- 实时价格
- 实时成交量
- 历史 K 线
- 24h 涨跌幅
- 可交易币种列表
- 最小下单精度 / 步长 / 最小名义价值

#### B. 真实账户与交易数据

- 账户总资产

- 已持仓币种
- 历史订单
- 当前挂单
- 实际成交记录

#### C. 币种池与交易对象

- 明确当前系统可以买哪些币种
- 明确币种白名单
- 明确每个币种是否允许自动交易
- 明确每个币种的风险等级

#### D. 图表与趋势分析

- K 线图
- 成交量图
- 策略信号标记
- 买入卖出点标记
- 当前持仓成本线
- 止损止盈线

#### E. 策略中心

- 展示当前有哪些策略
- 每个策略做什么
- 每个策略适用于什么周期
- 当前是否启用
- 当前参数是什么
- 当前哪些币种挂在这个策略上

#### F. 实盘执行

- 真实下单
- 真实撤单
- 同步真实订单状态
- 同步真实成交结果
- 明确当前订单来自哪个策略、哪个信号、哪个币种

#### G. 风控

- 单币最大仓位
- 总仓位上限
- 单日最大亏损
- 单次交易最大风险
- 黑名单币种
- API 故障熔断
- 策略熔断
- 全局停止开关

#### H. 复盘与记录

- 每笔交易为什么买
- 为什么卖
- 来自哪个策略
- 当时趋势状态是什么
- 收益如何
- 是否符合策略预期

## 5. 波段实盘阶段建议支持的第一批策略

当前阶段不应该一上来做很多策略，而应该先做少量、可解释、容易复盘的策略。

建议第一批只做 2 到 3 套：

### 5.1 趋势突破策略

适合：

- 强势币种
- 波段上行或下行切换阶段

核心逻辑：

- 价格突破近期区间高点
- 成交量放大
- 中周期趋势方向一致

要展示给用户的信息：

- 为什么触发突破
- 突破区间位置
- 当前止损位
- 当前目标位

### 5.2 趋势回踩策略

适合：

- 已形成中期趋势的币种
- 回调后再次延续

核心逻辑：

- 趋势方向已经明确
- 价格回踩均线或关键支撑
- 回踩结束后重新放量转强

要展示给用户的信息：

- 当前趋势是否有效
- 回踩深度是否过大
- 当前是否重新确认

### 5.3 相对强弱轮动策略

适合：

- 一组候选币之间的轮动选择

核心逻辑：

- 比较多个候选币的趋势强度
- 选择更强的币
- 强度衰减后调仓或退出

要展示给用户的信息：

- 当前币种池排名
- 轮入原因
- 轮出原因

用途：

- 读取账户
- 同步订单
- 真实下单

你需要准备：

- 1 组主账号现货 API Key / Secret

建议：

- 单独创建一组只给 Quant 使用的 API Key

## 8. 系统架构建议

## 8.1 顶层结构

```text
Binance Market Data
  -> Data Ingestion
  -> Feature / Indicator Layer
  -> Strategy Engine
  -> Signal Store
  -> Risk Engine
  -> Execution Orchestrator
  -> Freqtrade / Binance
  -> Orders / Trades / Positions / Balances
  -> Monitoring / Review / WebUI
```

## 8.2 模块拆分

### A. Data 模块

负责：

- 拉历史 K 线
- 拉实时价格
- 拉 24h ticker
- 拉订单簿摘要
- 维护币种元数据

输出给：

- 图表
- 指标计算
- 策略引擎

### B. Feature / Indicator 模块

负责：

- EMA / MA
- ATR
- RSI
- MACD
- 成交量特征
- 波动率
- 相对强弱评分

输出给：

- 策略引擎
- 图表叠加层

### C. Strategy 模块

负责：

- 波段策略定义
- 参数管理
- 币种筛选
- 信号生成

输出给：

- `signals`

### D. Risk 模块

负责：

- 头寸控制
- 单币风险控制
- 日内亏损限制
- 异常熔断
- API 故障处理

输出给：

- `risk_events`
- 执行允许 / 拒绝结果

### E. Execution 模块

负责：

- 把策略信号变成执行动作
- 调用 Freqtrade
- 同步订单和成交

### F. Review / Analytics 模块

负责：

- 收益统计
- 按策略统计
- 按币种统计
- 交易复盘
- 信号命中率分析

### G. WebUI 模块

负责：

- 实时工作台
- 图表
- 策略中心
- 仓位 / 订单 / 风险 / 复盘
