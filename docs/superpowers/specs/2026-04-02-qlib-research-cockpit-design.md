# Qlib 统一研究驾驶舱设计

## 背景判断

`Qlib` 这条线已经完成了两步：

- 最小训练、最小推理和结果展示
- 研究结果以软门控方式进入策略判断

但现在研究结果还是分散在几个页面里：

- 市场页主要看价格和推荐策略
- 单币图表页能看到研究解释，但没有和信号点、止损参考放在同一条判断链上
- 策略页能看到研究门控，但和市场页、图表页使用的语言还不统一

这会带来两个问题：

- 用户在三个页面里看到的是三套说法，不容易形成连续判断
- 后续别的 session 接 `Freqtrade` 时，研究层展示口径容易再次分叉

所以这一步不再做零散补丁，而是把研究层收敛成一套统一研究驾驶舱。

## 总体结论

推荐方案固定为：

- **统一研究摘要，页面按层消费**

也就是：

- 后端先生成一套统一研究展示结构
- 市场页消费简版摘要
- 单币图表页消费完整版摘要
- 策略页复用同一套字段，不再各自解释

这一步不新增独立新页面，而是在现有三个页面里统一表达。

## 设计目标

这次统一研究驾驶舱，只回答 5 个问题：

- 研究层当前站哪边
- 当前更适合哪套策略
- 这个判断强不强
- 关键证据是什么
- 现在下一步该做什么

统一后，三个页面都围绕同一组信息展开：

- `research_bias`：研究倾向，`bullish / bearish / neutral / unavailable`
- `recommended_strategy`：更适合的策略，`trend_breakout / trend_pullback / none`
- `confidence`：判断强度，`high / medium / low`
- `research_gate`：研究门控结果，继续沿用当前软门控状态
- `primary_reason`：当前主原因
- `research_explanation`：研究解释摘要
- `signal_count`：当前图表上的策略信号点数量
- `entry_hint`：最近入场参考
- `stop_hint`：最近止损参考
- `model_version`：模型版本
- `generated_at`：研究结果生成时间

## 页面分工

### 1. 市场总览页

市场页负责回答：

- 先看哪个币
- 它更偏哪套策略
- 研究层现在站哪边

这一页只用简版摘要，重点展示：

- 研究倾向
- 推荐策略
- 判断强度
- 主原因
- 模型版本

目的不是讲完整理由，而是让用户快速决定先点进哪个币种。

### 2. 单币图表页

单币页负责回答：

- 为什么这样判断
- 现在图表上最重要的信号点在哪里
- 失效位在哪里
- 下一步该做什么

这一页使用完整版摘要，把下列内容放到同一条判断链上：

- 研究倾向
- 推荐策略
- 研究门控
- 判断强度
- 主原因
- 研究解释
- 信号点数量
- 最近入场参考
- 最近止损参考
- 下一步动作

这里是统一研究驾驶舱的主解释页面。

### 3. 策略中心页

策略页负责回答：

- 这套策略现在能不能放行
- 研究层是在支持、压低还是观望

这一页保留当前软门控能力，但字段口径和市场页、单币页保持一致，避免用户在不同页面看到不同命名和不同解释方式。

## 后端拆法

后端不让前端自己拼判断，而是集中生成统一研究摘要。

建议拆成两层：

- 简版摘要：给市场总览页
- 完整版摘要：给单币图表页和策略页

实现上尽量复用现有服务：

- `research_service`
  - 继续负责读取研究结果
- `strategy_workspace_service`
  - 继续负责策略页聚合，但改为复用统一摘要字段
- `market_service`
  - 增加市场页和单币页需要的统一研究摘要生成

关键原则：

- 研究层展示逻辑尽量集中在后端
- 页面只负责渲染，不各自推导结论
- 软门控规则只有一套，不在不同页面重复实现

## 降级规则

统一研究驾驶舱必须允许部分能力缺失，但不能静默失败。

### 1. 研究不可用

- `research_bias` 统一返回 `unavailable`
- 页面仍显示原策略判断
- 页面文案明确提示“研究结果暂不可用”

### 2. 研究分数异常

- 按研究不可用处理
- 不参与门控
- 不让异常值直接进入页面判断

### 3. 图表数据不足

- 不硬算信号点和止损线
- 对应字段明确返回“数据不足”或空提示

### 4. 当前币种没有研究结果

- 明确返回“该币种暂无研究结论”
- 不拿别的币种结果填充

## 推荐方案与备选方案

### 方案 1：统一研究摘要，页面按层消费

这是推荐方案。

优点：

- 三个页面说同一种语言
- 研究层规则集中在后端，后续更稳
- 适合当前多 session 协同
- 不会和 `Freqtrade` 执行层开发互相打架

缺点：

- 需要补一层新摘要结构
- 需要同时改后端和三个页面

### 方案 2：只在前端补统一展示

不推荐。

问题：

- 前端会自己拼判断
- 三个页面容易再次分叉
- 后续规则变化时维护成本更高

### 方案 3：直接新增独立研究驾驶舱页面

当前阶段不推荐。

问题：

- 范围变大
- 和现有市场页、图表页、策略页形成重复入口
- 不利于先把现有主路径做顺

## 实施边界

这次只做第一阶段：

- 统一研究摘要结构
- 接到市场页、单币图表页、策略页
- 完成失败测试、实现、review、真实页面验证

这次不做：

- 新增独立研究驾驶舱页面
- 复杂图形化面板
- 自动调参
- 新执行链路
- 脱离现有页面的信息重组

## 验证原则

先写失败测试，再写实现，最后做真实验证。

至少覆盖：

- 后端摘要字段完整
- 页面都消费统一字段
- 缺失和异常场景能正确降级
- 市场页、单币页、策略页真实返回内容与关键展示一致

## 对多 session 协同的影响

这次统一的只是研究展示层，不改变执行层边界。

所以：

- 我这边继续负责 `Qlib` 研究层和展示口径
- 另一个 session 可以继续推进 `Freqtrade` 执行层
- 双方通过统一契约衔接，不直接互改对方核心逻辑
