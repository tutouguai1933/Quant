# Quant 前端工作台重构设计

这份文档只回答三件事：

- 为什么现在的前端体验不好
- 应该重构成什么样
- 多因子量化流程应该怎样显化到系统里

## 1. 背景

当前系统已经有了不少能力：

- 数据工作台
- 特征工作台
- 研究工作台
- 回测工作台
- 评估与实验中心
- 执行与自动化工作台

但前端当前的主要问题已经不是“功能不够”，而是“默认页面展开的信息过多”。

典型表现：

- 每一页都同时在讲状态、配置、解释、动作、历史、异常
- 用户第一眼找不到“现在最该看什么”
- 用户想执行一个动作时，必须先读大量说明文字
- 研究、决策、执行、运维之间的边界不够清楚
- 默认视图像“开发调试面板”，不像“日常使用工作台”

这会导致两个直接后果：

1. 新用户或非技术用户很难建立操作主线  
2. 老用户也很难在高频使用时快速定位重点

## 2. 目标

这次重构的目标不是加更多页面，而是把“默认视图”从信息堆叠改成“摘要优先、细节按需打开”。

本次重构要达到：

- 默认只显示当前最重要的信息
- 一个页面只回答一个主问题
- 一个页面只保留一个主动作区
- 详细解释、历史、配置进入抽屉、弹窗或二级详情
- 让首页可以覆盖大部分日常使用路径
- 把多因子量化流程明确显化出来，而不是只保留“特征配置”

## 3. 非目标

这次重构不做这些事情：

- 不重写整套后端研究引擎
- 不马上把系统改成完整多资产组合交易平台
- 不一次性移除所有旧页面
- 不先做视觉换肤再谈信息架构

重点先放在：

- 信息架构
- 主线动线
- 详情收纳方式

## 4. 可选方案

### 方案 A：继续保留现有页面，只补折叠区

优点：

- 改动小
- 风险低

缺点：

- 根问题没有解决
- 页面主问题仍然不清楚
- 用户还是需要先理解很多卡片的含义

### 方案 B：主工作台 + 深度工作台 + 抽屉 / 弹窗

优点：

- 既保留现有能力，又能把默认视图收窄
- 最适合当前已经有较多页面和较多后端能力的系统
- 能同时解决“入口太散”和“页面信息过多”

缺点：

- 需要先统一信息架构
- 会涉及多个主页面重构

### 方案 C：改成向导式单流程产品

优点：

- 新用户更容易理解

缺点：

- 不适合高频使用
- 不适合研究、盯盘、运维同时进行的场景

## 5. 结论

采用 **方案 B**。

也就是：

- 保留当前系统的大能力边界
- 重构默认页面
- 新增一个真正的主工作台
- 把细节尽量收进抽屉、弹窗和详情页

## 6. 设计原则

### 6.1 摘要优先

默认页面只展示：

- 当前状态
- 当前结论
- 当前风险
- 当前下一步动作

### 6.2 一页一问

每个主页面只回答一个核心问题。

例如：

- 首页：现在整个系统处于什么状态，下一步该做什么
- 因子页：当前因子体系长什么样，哪些因子有效
- 研究页：当前研究在跑什么，用了什么配置
- 决策页：为什么推荐 / 为什么淘汰 / 当前该推进谁
- 执行页：执行链是否安全，当前订单和接管状态如何

### 6.3 细节按需展开

这些内容不再默认铺开：

- 大表格
- 历史列表
- 详细配置
- 门控细节
- 执行差异明细
- 因子相关矩阵
- 实验对比明细

它们统一进入：

- 右侧抽屉
- 底部详情面板
- 弹窗
- 二级页

### 6.4 动作优先于解释

页面应该先告诉用户：

- 现在能不能继续
- 下一步要去哪
- 当前最值得点击哪个按钮

然后再解释为什么。

### 6.5 统一状态语言

所有页面统一使用这四类状态：

- 正常
- 运行中
- 阻塞
- 需人工处理

不要让不同页面各自发明自己的状态词。

## 7. 目标结构

前端未来收成：

### 一级工作区

- `/`：主工作台
- `/features`：因子工作台
- `/research`：研究工作台
- `/evaluation`：决策工作台
- `/strategies`：执行工作台
- `/tasks`：运维工作台

### 二级工具页

- `/market`
- `/balances`
- `/orders`
- `/positions`
- `/risk`

二级工具页继续保留，但不再作为主链入口。

## 8. 主工作台设计

主工作台是后续最重要的新入口。

它默认只保留 6 个核心区块。

### 8.1 当前推荐

回答：

- 当前最值得关注的币是谁
- 当前处于研究 / dry-run / live 哪一层
- 主推荐理由是什么

默认显示：

- 币种
- 当前阶段
- 推荐摘要
- 一个主按钮：去决策页

### 8.2 当前研究状态

回答：

- 研究有没有在跑
- 当前跑到哪一阶段
- 预计还要多久

默认显示：

- 当前动作
- 当前阶段
- 进度
- 一个主按钮：去研究页

### 8.3 当前执行状态

回答：

- 执行器是否可用
- 当前是否处于 dry-run / live
- 最近订单和持仓有没有变化

默认显示：

- 执行模式
- 执行器连接状态
- 最近订单摘要
- 一个主按钮：去执行页

### 8.4 当前风险与告警

回答：

- 当前最会阻塞下一步的风险是什么

默认显示：

- 头号告警
- 风险等级
- 人工接管状态
- 一个主按钮：去运维页

### 8.5 当前下一步动作

回答：

- 现在最该做的动作是什么

默认显示：

- 动作标题
- 动作原因
- 目标页面
- 一个主按钮：立即前往

### 8.6 最近结果回看

回答：

- 最近一轮研究 / 执行 / 复盘的最终结果是什么

默认显示：

- 最近工作流来源
- 最近结果摘要
- 一个主按钮：去详情抽屉

## 9. 因子工作台设计

这是当前系统最缺少“显化”的部分，也是你给的 PPT 最值得吸收的部分。

现在的 `/features` 更像“特征配置页”，未来要升级成真正的“因子工作台”。

### 9.1 因子工作台要回答的问题

- 当前因子体系有哪些
- 哪些因子正在启用
- 哪些因子有效
- 哪些因子高度重复
- 某个候选为什么会得到当前总分

### 9.2 默认视图只保留 5 块

#### 因子分类总览

- 趋势
- 动量
- 震荡
- 成交量
- 波动率
- 其他扩展因子

#### 当前启用因子

- 当前主判断因子
- 当前辅助因子
- 当前权重

#### 因子有效性摘要

- IC
- 分组表现
- 稳定性
- 是否通过

#### 因子冗余摘要

- 高相关因子数量
- 已合并 / 已剔除数量
- 当前保留集大小

#### 总分解释入口

- 当前推荐币的分数由哪些因子构成

### 9.3 详情层

点击因子后再看：

- 时间序列
- IC 曲线
- 分组收益曲线
- 单调性
- 稳定性
- 与其他因子的相关矩阵

### 9.4 对 PPT 的适配

PPT 讲的是股票多因子选股，我们在当前系统里先适配成：

- 候选币多因子筛选
- 候选池打分
- `dry-run / live` 候选篮子

也就是先做：

- 多因子候选排序

而不是一步到位做：

- 完整股票组合管理系统

## 10. 研究工作台设计

未来的 `/research` 只回答：

- 当前研究在跑什么
- 当前配置是什么
- 当前产物是什么

默认页不再展开整片大配置区。

### 10.1 默认显示

- 当前模板
- 当前模型
- 当前标签方式
- 当前数据范围
- 当前运行状态
- 当前产物摘要

### 10.2 抽屉内容

- 完整参数配置
- 模型目录
- 标签目录
- 切分细节
- 当前配置快照

### 10.3 弹窗内容

- 最近实验对比
- 模板差异
- 训练产物详情

## 11. 决策工作台设计

未来的 `/evaluation` 要真正成为“决策页”，而不是“所有评估信息罗列页”。

默认只回答：

- 为什么推荐
- 为什么淘汰
- 当前卡在哪
- 当前该推进谁
- 下一步该去哪

### 11.1 默认显示

- 当前仲裁结论
- 当前推荐候选
- 当前主要阻塞
- 当前下一步动作
- 推荐摘要
- 淘汰摘要

### 11.2 抽屉内容

- 门控细节
- 实验差异
- 执行差异
- 历史实验
- 对齐差异账本

## 12. 执行与运维工作台设计

`/strategies` 和 `/tasks` 不再试图把所有东西都铺开。

### 12.1 `/strategies`

默认回答：

- 当前执行器是否可用
- 当前候选是否可推进
- 当前执行模式是什么
- 当前账户收口是否正常

### 12.2 `/tasks`

默认回答：

- 当前自动化状态
- 当前告警
- 当前人工接管状态
- 当前恢复建议

### 12.3 详情层

这些都进入折叠区或抽屉：

- 告警历史
- 恢复清单
- 失败规则
- 调度顺序
- 执行细节
- 同步失败明细

## 13. 导航重构

当前主导航项目太多，且主线和工具页混在一起。

未来建议：

### 主导航

- 总览
- 因子
- 研究
- 决策
- 执行
- 运维

### 工具入口

- 市场
- 余额
- 持仓
- 订单
- 风险

工具入口不再和主导航争抢第一层注意力。

## 14. 交互规范

### 14.1 默认卡片只保留三段

- 标题
- 一句话摘要
- 一个主动作

### 14.2 所有“查看详情”统一形式

- 右侧抽屉优先
- 大表格和历史列表再用弹窗或二级页

### 14.3 所有按钮都必须有反馈

- 点击中
- 成功
- 失败
- 跳转后反馈

### 14.4 所有异常都必须有人话说明

不要只显示：

- unavailable
- fetch failed
- blocked

要直接说明：

- 卡在哪
- 是否能继续
- 下一步该去哪

## 15. 与当前系统的关系

这次重构不是推倒重来，而是：

- 保留当前后端契约
- 先重排前端入口
- 先收默认视图
- 再把细节移入抽屉 / 弹窗

也就是说：

- 先改信息架构
- 再改页面结构
- 最后再统一视觉层

## 16. 成功标准

重构完成后，应该满足：

1. 首页可以覆盖大部分日常判断  
2. 用户不需要翻很多页就能知道“现在该做什么”  
3. 因子主线被明确显化出来  
4. 研究、决策、执行、运维四条线边界清楚  
5. 默认页面不再像调试面板
